7. GITHUB ACTIONS CI/CD — AUTHORITATIVE GATE DESCRIPTION
----------------------------------------------------------------------
Sovereign Forge operates two GitHub Actions workflows. These workflows
are the primary quality enforcement mechanism for the project. Unlike
documentation-only quality gates, these workflows are AUTOMATED and
BLOCKING: no code can be merged to main without a passing ci.yml run.

WORKFLOW 1: ci.yml (Continuous Integration)
Location:   .github/workflows/ci.yml
Trigger:    Every push to any branch; every pull request targeting main
Purpose:    Automated test execution, lint, and integration validation

ci.yml Pipeline Steps (logical sequence):
  Step 1 — Checkout
    Action: actions/checkout@v3
    Purpose: Clone repository at the triggering commit

  Step 2 — Python Setup
    Action: actions/setup-python@v4
    Purpose: Pin Python version matching requirements.txt

  Step 3 — Dependency Installation
    Command: pip install -r requirements.txt
    Purpose: Install all project dependencies in a clean environment

  Step 4 — Environment Configuration
    Mechanism: GitHub Secrets injection
    Variables: MOCK_AI=true (forces mock mode for CI), any required
               test environment variables
    Security: No API keys committed to source; all credentials are
              GitHub repository secrets referenced by workflow name

  Step 5 — Test Execution
    Command: pytest tests/ -v
    Scope:   All 13 test files in tests/ directory
    Failure: Any single test failure causes the entire CI run to FAIL
             and blocks merge

  Step 6 — Coverage Reporting (if configured)
    Purpose: Track test coverage percentage over time

Gate Enforcement:
  Branch protection rules on main require ci.yml to pass before any
  merge. This means:
    - Every feature branch must pass all 13 test files
    - No manual override is possible without administrator intervention
    - The quality gate is enforced by GitHub infrastructure, not by
      developer discipline alone

This is the defining DevOps-maturity characteristic of Sovereign Forge.


WORKFLOW 2: release.yml (Release Automation)
Location:   .github/workflows/release.yml
Trigger:    Push of a version tag (e.g., v1.0.0, v2.0.0)
Purpose:    Automated release packaging, artifact publishing, and
            version tagging

release.yml Pipeline Steps (logical sequence):
  Step 1 — Checkout with Full History
    Purpose: Access git history for changelog generation

  Step 2 — Python Setup and Dependency Installation
    Purpose: Consistent build environment

  Step 3 — Pre-Release Security Check
    Purpose: Validate no secrets are present in release artifacts
    Complement: deep_clean.ps1 should have been executed locally
                before pushing the release tag

  Step 4 — Build Artifacts
    Purpose: Package the application (Docker image, distribution
             archive, or both)

  Step 5 — GitHub Release Creation
    Action: actions/create-release or softprops/action-gh-release
    Purpose: Create a GitHub Release with version tag, release notes,
             and attached artifacts

  Step 6 — Artifact Attachment
    Purpose: Attach build artifacts (Docker image digest, archive)
             to the GitHub Release for traceability

Release Governance:
  - Every GitHub Release corresponds to a release.yml execution
  - Release tags follow semantic versioning (MAJOR.MINOR.PATCH)
  - Release cadence target: minimum 1 release per sprint (see KPI-008)
  - Manual releases are not permitted; all releases go through the
    pipeline to ensure artifact signing and version tagging integrity

10. AZURE AI EVALUATION SDK INTEGRATION
----------------------------------------------------------------------
The Azure AI Evaluation SDK is integrated into Sovereign Forge as a
post-generation quality scoring layer. This is a unique capability
within the portfolio.

Integration Points:
  evaluation/collect_responses.py
    - Connects to the running API
    - Submits queries from data/test_queries.json (90 test cases
      across chat / command / philosophy categories)
    - Collects AI responses and writes them to a structured dataset

  evaluation/run_evaluation.py
    - Loads collected responses
    - Invokes Azure AI Evaluation SDK evaluators:
        AzureOpenAIRelevanceEvaluator
        AzureOpenAICoherenceEvaluator
        AzureOpenAIGroundednessEvaluator
    - Each evaluator scores responses on a 1–5 integer scale
    - Computes composite score as weighted average
    - Writes results to evaluation/eval_results.json

  evaluation/HANDOFF_TO_CLAUDE.ipynb
    - Jupyter notebook for interactive evaluation review and handoff

Scoring Rubric:
  Score 1 — Completely irrelevant / incoherent / ungrounded
  Score 2 — Mostly irrelevant / weak coherence / poor grounding
  Score 3 — Partially relevant / moderate coherence / partial
             grounding
  Score 4 — Mostly relevant / good coherence / well grounded
  Score 5 — Fully relevant / highly coherent / fully grounded

Target: Composite score > 4.2/5.0

Current Status:
  Pipeline code is complete. eval_results.json population with live
  Azure scoring data is a Phase 4 deliverable (RISK-010 mitigation
  is in place via mock mode validation).

----------------------------------------------------------------------
Sovereign Forge is the only project in the portfolio with dedicated
production cleanup scripts. These scripts are part of the security
gate (GATE 4) and must be executed before any release tag is pushed.

scripts/deep_clean.ps1
  Purpose:    Comprehensive pre-release cleanup
  Actions:
    - Remove all __pycache__ directories and .pyc files
    - Remove pytest temporary files and .pytest_cache
    - Remove .env files and any local credential files
    - Remove build/, dist/, *.egg-info/ directories
    - Clear evaluation/eval_results.json cached partial runs
    - Remove any IDE-generated temporary files
    - Validate that no credential patterns exist in tracked files
  When:       Before pushing any release tag
  Owner:      Release engineer

scripts/cleanup_artifacts.ps1
  Purpose:    Targeted artifact removal for intermediate cleanup
  Actions:
    - Remove Docker build cache artifacts
    - Remove log files from local test runs
    - Clear temporary API response caches
  When:       After each development session, before commit

Artifact Governance Rules:
  1. No API keys, connection strings, or secrets may be committed
     to the repository at any time.
  2. All credentials are injected via environment variables or GitHub
     Secrets.
  3. deep_clean.ps1 must complete without errors before release.yml
     is triggered.
  4. The .gitignore and .dockerignore files are the first line of

  1. Create branch from main
  2. Implement change
  3. Verify ci.yml passes on branch
  4. Open pull request with description referencing issue template
  5. Peer review (minimum 1 approver)
  6. Merge after CI green gate confirmed

MAJOR CHANGES (new features, architecture changes, new phases):
  1. Update SOVEREIGN-SDLC-MASTER-001 with phase or scope change
  2. Update SOVEREIGN-RISK-REGISTER-001 if new risks introduced
  3. Update SOVEREIGN-PRODUCT-BACKLOG-001 with new/modified stories
  4. Create feature branch
  5. Implement change with test coverage
  6. Verify all 13 test files pass in ci.yml
  7. Execute cleanup scripts
  8. Open pull request with full description
  9. Architect + Security Officer review
  10. Merge after all gates clear

RELEASE PROCESS:
  1. Confirm all 4 quality gates are green

  3. Execute cleanup_artifacts.ps1

  5. release.yml triggers automatically
  6. Verify GitHub Release artifact is published
